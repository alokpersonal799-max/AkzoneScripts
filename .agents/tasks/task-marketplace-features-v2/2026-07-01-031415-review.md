# Marketplace features: invoices, changelogs, custom pages, popup (v2)

Second-pass review after fixes for duplicate form fields, popup caching, and invoice rate limiting. The three actionable issues from v1 (issues #2, #3, #5) have been addressed. Issues #1 and #4 (stored XSS from admin-authored HTML content and the preview route) were accepted as documented admin-trust decisions and remain unchanged.

**Watch for:** Popup countdown timer leaks a `setInterval` that never clears (confirmed). Invoice controller allows PDF generation for non-completed orders, exposing partial/pending order data as a branded invoice (likely). The `x-bind:name` fix nullifies inactive field names but doesn't prevent the browser from submitting those inputs with empty-string values for fields that have no `name` attribute, which may interact unexpectedly with Laravel's nullable validation (possible).

**Verdict**: APPROVED

## High-level view

The v1 fixes are cleanly resolved. Dynamic `:name` binding ensures only the active popup mode's fields carry a name attribute on submit. The popup product query is cached with a 5-minute TTL. The invoice route carries `throttle:10,1`.

The remaining concerns are minor: a timer interval that never clears (cosmetic memory leak in a popup that persists until page navigation), and an authorization gap where non-completed orders can still produce branded invoices.

## Popup countdown interval leak

In `partials/popup.blade.php`, the offer-mode countdown timer initializes an interval that runs indefinitely:

```javascript
init() { this.tick(); setInterval(() => this.tick(), 1000); }
```

The `close()` method on the parent popup component clears its own auto-close timeout but has no reference to this inner countdown interval. Since the countdown lives in a nested `x-data` scope inside the popup card, it continues ticking after the overlay is dismissed. Alpine's `x-show` does not destroy scopes (only `x-if` does), so the interval persists until full page navigation.

## Invoice authorization gap

```php
abort_unless($order->user_id === auth()->id(), 403);
```

This doesn't gate on order status. A user with a pending/failed order can hit `/orders/{id}/invoice` and receive a PDF with the store's branding, logo, and "Thank you for your purchase!" message — despite never completing payment. The UI only shows the invoice button when `$order->isCompleted()`, but the route itself doesn't enforce this, so direct URL access bypasses the UI gate.

## Note on v1 fix scope

The `setting()` calls in the popup partial were never an N+1 problem — `Setting::get()` reads from a `Cache::rememberForever` store that loads all settings in one query. The actual per-request cost was the `Product::where()->first()` call, which the `Cache::remember` fix correctly addresses.
