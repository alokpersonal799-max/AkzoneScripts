# Four marketplace features: PDF invoices, product changelogs, custom pages redesign, promotional popup

This change adds four buyer/admin-facing features to the Laravel digital marketplace: PDF invoice downloads via barryvdh/laravel-dompdf, a product changelog/version history system, a revamped custom pages editor supporting raw HTML with pre-designed templates and preview, and a configurable promotional popup with three display modes. The implementation is convention-following Laravel work — migrations, models, Eloquent relationships, Blade views with Alpine.js interactivity. Each feature is self-contained with no complex inter-dependencies.

**Watch for:** Stored XSS via unescaped HTML page content (confirmed). Duplicate form field names in the popup admin form cause data loss depending on which mode is submitted (confirmed). The popup fires a DB query on every guest page load with no caching (confirmed). No rate limiting on CPU-intensive PDF invoice generation (likely).

## High-level view

The PDF invoice system is a single controller with auth check, eager-loaded order items, and a styled Blade template rendered via DomPDF. There's no rate limiting on the endpoint, and DomPDF is CPU-intensive — a user could hammer their own invoice URL and stress the server.

The custom pages redesign introduces a `content_type` column toggling between plain text (escaped) and raw HTML (unescaped). This is the change's most significant security surface — any admin can inject arbitrary JavaScript that executes in visitor sessions. The template system is purely client-side (JS string templates in a Blade partial), and preview uses a POST form submission to a new route that renders the page layout without saving.

The promotional popup is the most complex addition. It supports three modes (message, product, offer-with-countdown), auto-close with a progress bar, localStorage-based frequency control, and an overlay dismiss. The admin form has a structural bug: multiple `<input name="popup_message">` and `<input name="popup_product">` elements exist across mode-specific fieldsets that are hidden with `x-show` (not removed from DOM), so browsers submit the wrong field values. The popup partial also runs an Eloquent query inline on every page load when enabled.

1. **Stored XSS in custom pages** — HTML content type renders `{!! $page->content !!}` without sanitization. Any admin can inject scripts that execute in visitor browsers. Add server-side HTML sanitization (e.g., HTMLPurifier) before storage, or accept this as admin-only trust and document the decision. (confirmed)

Use Alpine to dynamically set the field name, or use a single shared field outside the mode-specific blocks. (confirmed)
3. **Popup DB query on every page load** — `partials/popup.blade.php` runs `Product::where(...)→first()` inline plus 10+ `setting()` calls. With popup enabled, every guest page load pays this cost. Wrap in a short-TTL cache or push the product data into a view composer. (confirmed)
4. **Preview route renders arbitrary HTML** — `POST /admin/pages/preview` renders any HTML passed in the request body using `{!! !!}`. While behind `auth` + `admin` middleware, a hijacked admin session could use this as an XSS vector against the admin themselves. Consider rendering preview client-side via iframe `srcdoc` instead. (likely)
5. **No rate limiting on PDF generation** — The invoice endpoint renders full HTML-to-PDF via DomPDF on each request. No throttle or caching protects against repeated requests. Add a rate limiter or cache the generated PDF for a short period. (likely)

## Stored XSS surface in custom pages

The `pages/show.blade.php` view renders HTML content with `{!! $page->content !!}` when `content_type` is `'html'`. There is no server-side sanitization — the `PageController::validatePage` method only validates `content_type` as `in:text,html` and accepts the content string as-is. The preview route exhibits the same pattern.

```php
// PageController.php - preview method
$page = new Page([
    'content' => $data['content'] ?? '',
    'content_type' => $data['content_type'] ?? 'text',
]);
```

```blade
{{-- pages/show.blade.php --}}
@if(($page->content_type ?? 'text') === 'html')
    <div class="page-html-content">
        {!! $page->content !!}
    </div>
```

A compromised admin account or a CSRF attack against the page editor would plant persistent XSS affecting all site visitors. The templates partial advertises "Full HTML/CSS/JS is supported" in the editor hint text, so this appears to be a deliberate feature — but it should be a conscious, documented trust decision.

## Duplicate form field names in popup admin

The promotions admin form contains mode-specific fieldsets hidden with `x-show` + `x-cloak`. Each mode section has its own `name="popup_message"` textarea and `name="popup_product"` select:

```blade
{{-- Message mode --}}
<textarea id="popup_message_msg" name="popup_message" ...>

{{-- Product mode --}}
<textarea id="popup_message_prod" name="popup_message" ...>

{{-- Offer mode --}}
<textarea id="popup_message_offer" name="popup_message" ...>
<select id="popup_product_offer" name="popup_product" ...>

{{-- Product mode also has --}}
<select id="popup_product_select" name="popup_product" ...>
```

Alpine's `x-show` hides elements visually but leaves them in the DOM. When the form submits, the browser serializes all form fields — including hidden ones. For duplicate names, browsers send the last value in document order. This means:
- If the admin is in "message" mode, the submitted `popup_message` value comes from the "offer" textarea (the last one in the DOM), which may be empty.
- If the admin is in "product" mode, the submitted `popup_product` comes from the "offer" select.

The result is silent data loss — the admin saves settings, but the persisted values don't match what was entered in the active mode panel.

## Popup inline query without caching

The `partials/popup.blade.php` view runs business logic inline:

```php
@php
    $popupProductId = (int) setting('popup_product', 0);
    $popupProduct = null;
    if (($popupMode === 'product' || $popupMode === 'offer') && $popupProductId) {
        $popupProduct = \App\Models\Product::where('id', $popupProductId)->where('status', 'published')->first();
    }
    $popupHash = substr(md5($popupMode . $popupHeading . $popupMessage . $popupProductId . $popupTimerUntil), 0, 10);
@endphp
```

This executes on every page load for every visitor when the popup is enabled. The multiple `setting()` calls (10+) likely each query the settings store, multiplied across all visitors. For a feature that shows identical content to every visitor, this should be cached — either via `Cache::remember()` wrapping the whole popup data assembly, or by moving it into a view composer with a short TTL.
